# T2 â€” Terrain Map (.T2)

FA_2.LIB contains 16 `.T2` files â€” one per theater. Each stores the terrain
grid data (height, color, surface type) for a campaign area. Referenced by
`.MM` theater files via the `map` keyword (e.g. `map apa.T2`).

## Tools


## File Layout

All multi-byte integers are little-endian.

### Header

| Offset | Size | Type | Description |
|--------|------|------|-------------|
| `0x00` | 4  | char[4] | Magic `BIT2` (ASCII, no null) |
| `0x04` | 60 | char[60] | Theater name, null-padded ASCII (e.g. "Panama") |
| `0x40` | 20 | ?   | **Unknown** (observed all-zero) |
| `0x54` | 12 | char[12] | PIC texture atlas reference, null-padded ASCII (e.g. "apa.PIC") |
| `0x60` | 4  | ?   | **Unknown** (observed 0) |
| `0x64` | 4  | u32 | Map dimension (tile columns = tile rows; 25 or 32) |
| `0x68` | 4  | ?   | **Unknown** (observed 0) |
| `0x6C` | 4  | u32 | Map dimension (duplicate of 0x64) |
| `0x70` | 8  | ?   | **Unknown** (observed all-zero) |
| `0x78` | 4  | u32 | Always 2048 |
| `0x7C` | 4  | u32 | stored_col_count Ă— 256 (stored_col_count â‰Ą dim; may pad to even column) |
| `0x80` | â€”  |     | Terrain data payload (structure below) |

### Map Sizes

`[0x7C]` encodes the **stored column count Ă— 256** â€” which may exceed `dim`
when the tile array is column-padded:

| Grid | Files | dim_x (`[0x64]`) | dim_y (byte `[0x7D]`) | tiles | File size |
|------|-------|-----------------|----------------------|-------|-----------|
| 32Ă—32 | 11 | 32 | 32 | 1024 | 199,829 B |
| 25Ă—26 | 4 (EGY/FRA/UKR/VLA) | 25 | 26 | 650 | 126,899 B |
| 25Ă—25 | 1 (TVIET) | 25 | 25 | 625 | 122,024 B |

When addressing tile (col, row): `tile_index = row Ă— dim_x + col`.

### Data Payload

Payload at offset `0x80` = **21-byte sub-header** + **N_tiles Ă— 195 bytes** of
tile data.

| Offset (from 0x80) | Size | Description |
|--------------------|------|-------------|
| `+0`  | 21  | Sub-header (partially decoded; see below) |
| `+21` | 195 | Tile 0 |
| `+216`| 195 | Tile 1 â€¦ (tiles stored row-major; tile_index = row Ă— stored_cols + col) |

### Sub-Header (21 bytes)

| Byte(s) | Observed values | Notes |
|---------|-----------------|-------|
| 0 | `0x00` | Always 0 |
| 1 | `0x20` (32Ă—32), `0x19` (25Ă—N) | Map x-dimension (= `dim` at header `0x64`) |
| 2â€“3 | `0x00 0x00` | Always zero |
| 4â€“16 | *class constant â€” see below* | Not per-theater; one fixed value per grid-size class |
| 17 | `0x95` | Always 0x95 = 149 = absolute file offset of first tile (0x80 + 21) |
| 18â€“20 | `0x00 0x00 0x00` | Always zero |

Bytes 4â€“16 are **class constants** â€” all theaters of the same grid-size class
share identical values. Full survey of all 16 theaters confirmed three distinct

| Grid class | Files | Bytes 4â€“16 (hex) |
|------------|-------|-------------------|
| 32Ă—32 | APA BAL CUB GRE IRA KURILE LFA NSK PGU SPA WTA (11 files) | `00 95 00 03 00 00 01 00 00 00 01 00 00` |
| 25Ă—26 | EGY FRA UKR VLA (4 files) | `00 15 E8 01 00 D0 00 00 00 C8 00 00 00` |
| 25Ă—25 | TVIET (1 file) | `00 55 D5 01 00 C8 00 00 00 C8 00 00 00` |

These values do not map cleanly to tile counts, file sizes, or simple dim

### Tile Layout (195 bytes = 65 Ă— 3-byte records)

Each tile's 195 bytes divide into **65 consecutive 3-byte records**:

| Record index | Role |
|--------------|------|
| 0 | Tile summary â€” representative sample for LOD/overview rendering (same structure as sub-records) |
| 1â€“64 | 8Ă—8 sub-tile grid, row-major (rec 1 = sub-row 0 col 0; rec 8 = sub-row 0 col 7; rec 9 = sub-row 1 col 0; â€¦) |

The summary record is **not** the NW corner (sub-record 1) and is **not** the
dominant surface class â€” survey of mixed-terrain tiles in UKR.T2 shows
disagreement in both cases. See Open Questions for the selection algorithm.

### 3-Byte Sub-Tile Record

| Byte | Field | Values / Notes |
|------|-------|----------------|
| 0 | Surface class | `0xFF` = water/ocean (all theaters); `0xD2` = land (APA, IRA, GRE, NSK, LFA â€” single land type); `0xD0`â€“`0xDA` = multiple land classes (EGY, UKR: sand, rocky, dunes, etc.); `0xC2`, `0xC4`â€“`0xC7` = land classes (TVIET: jungle, paddy, savanna, etc.) |

| 2 | Texture variant | 0â€“31; selects a sub-texture within the PIC atlas for this surface class |

## Engine Notes

### Surface Class â†’ Palette Mapping (`Remap` pipeline)

The surface class byte (byte 0 of each 3-byte sub-tile record) is never used as
a raw palette index. At scene init, `@DoSetTmapRemaps@0` (`0x004cc518`) builds
a 256-byte precomputed cache:

```c
// pseudocode â€” restored by fixing Remap()'s signature in Ghidra
for (uint i = 0; i < 256; i++)
    _tmapRemapTable[i] = Remap((byte)i);
```

The terrain renderer indexes into `_tmapRemapTable[surface_class]` per-polygon
for speed rather than calling `Remap()` directly.

### `Remap()` â€” `0x004cc44c`

`Remap(byte surface_class)` maps a surface class to a final 8-bit palette index
under the current atmosphere state. Its pipeline (decompiled after signature
fix):

**Stage 1 â€” Extended surface class redirect**

If the incoming value is > 255 (i.e. an extended surface class passed via full
EAX), the function indexes into `DAT_0055b9e0`, a `uint16_t[]` redirect table:

```
palette_index = DAT_0055b9e0[surface_class]   // ushort lookup
if palette_index > 255: return palette_index directly (bypass atmosphere)
```

Surface classes 0â€“255 pass through this stage unchanged.

**Stage 2 â€” Atmosphere / lighting chain** (gated by `_effects` bits)

Applied only when `_effects & 0x8` is set:

| Step | Condition | Operation |
|------|-----------|-----------|
| Shade table | `unaff_EBX != 0` (caller-set flag) | `idx = _currentShadeTable[idx]` |
| Blend tables | `_effects & 0x10` | `idx = DAT_005843c8[DAT_005843c4[idx]]` |
| Tint table | always (within effects block) | `idx = _currentTintTable[idx]` |
| Brightness | `idx != 0xFF` | `idx += _globalColorAdd` |

**Stage 3** â€” return the final `byte` palette index.

### Globals referenced

| Symbol | Role |
|--------|------|
| `_tmapRemapTable` | 256-byte precomputed cache; rebuilt at scene init by `@DoSetTmapRemaps@0` |
| `DAT_0055b9e0` | `uint16_t[]` extended surface class redirect table (surface class > 255 path) |
| `_effects` | Rendering effects bitfield; bit 3 gates atmosphere chain, bit 4 gates blend layer |
| `_currentShadeTable` | Secondary atmosphere layer pointer; applied when `unaff_EBX != 0` |
| `_currentTintTable` | Primary atmosphere layer pointer; always applied within the effects block |
| `DAT_005843c4` / `DAT_005843c8` | Additional atmosphere blend layer pointers (double-pass blend) |
| `_globalColorAdd` | Global brightness/gamma shift; added to final index unless it is `0xFF` |
| `unaff_EBX` | Implicit caller-set flag (EBX) gating the shade table step; semantics unknown |

The `_currentShadeTable`, `_currentTintTable`, `DAT_005843c4`, and
`DAT_005843c8` are the same atmosphere layer globals populated by
`ParseLayerFile` / `SetActiveLayerByAngle` (see
[architecture.md](../architecture.md) â€” Sky & Atmosphere subsystem).

### Relation to `.MM` tmap coordinates

`MISSIONTextProc` is the MM file text parser (keyword dispatch loop). The
`tmap` keyword handler (confirmed from decompile) reads **4 Ă— s16** values per
entry (format `tmap %d %d %d %d`), bounded to 3500 entries (counter
`tlistSize`), stored into arrays at `tlist`/`DAT_005733e4`. A second form
`tmap_named` reads a name string + 2 Ă— s16, stored as a 53-byte struct at
`tdic + i Ă— 0x35`. The `tdic` keyword reads 1 Ă— u32 + 32 Ă— u8 into the same
struct at offset `+0x0D` (max 300 entries, counter `tdicSize`).

Approximate tile-index formula (based on observed `tmap` values in MM files):

```
T2_tile_col = tmap_col / 8
T2_tile_row = tmap_row / 8
tile_index  = T2_tile_row Ă— dim_x + T2_tile_col
```

## Open Questions

### 1. Sub-header bytes 4â€“16 (class constants)

The T2 loader was not found in analyzed code: `_G_TileInit` (`0x447a40`) is
trivial (allocates a bitmap buffer only), the sub-header constant scan
(`findFunctionsReadingOffsets` for `0x95`, `0x80`, `195`, `21`) found no
T2-specific functions, and the BIT2 magic string is confirmed absent from
The game executable. The T2 file is loaded through the generic MM/LIB pipeline without
explicit byte-offset reads of the sub-header constants.

The constants at bytes 4â€“16 are most likely world-space coordinate bounds
passed as a block to `@G_Tile@32` as parameters rather than being read
field-by-field at load time. The `@G_Tile@32` renderer uses world-space
coordinates extensively (viewer Y, aspect ratios, `_worldCenter`, `_mapScale`)
â€” the sub-header block is a candidate coordinate-system initializer.

Next step: instrument a live the game executable session â€” break at `@G_Tile@32` entry and
inspect the argument registers against the known 13-byte constants to see which
parameter holds each value.

*Status: open â€” re-gameplay (#56)*

### 2. Tile summary record (record 0) selection algorithm

The renderer is too complex for static analysis alone: `@G_Tile@32`
(`0x447aa5`) decompiles to ~880 lines of scanline terrain rendering code
(perspective projection, haze, blending). The function receives pre-loaded tile
data via parameters; it does not parse the 195-byte tile structure internally.
The constant `0xC3` (195) does not appear in the tile-cluster code â€” tile
stride is likely computed via a dimension variable, not a literal.

Record 0 is almost certainly **pre-baked in the file** (authored, not
engine-computed at load time) â€” the simplest explanation consistent with the
absence of any averaging/sampling code in the analyzed rendering path. Live
confirmation is straightforward: modify record 0 of a single tile in a T2 file
(change its surface class and elevation) while leaving records 1â€“64 unchanged;
if the terrain overview (minimap or distant LOD view) changes, record 0 is
pre-baked. If engine-computed is suspected instead: trace `FUN_004aacfe`
(`0x4aacfe`) â€” the sole caller of `@G_Tile@32` â€” to see if it pre-processes
tiles before passing them to the renderer.
